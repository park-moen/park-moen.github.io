## 1단계: `next/dynamic`이란 무엇인가

### `React.lazy` + `Suspense`의 SSR 래퍼

`React.lazy`는 Server Component에서 사용할 수 없습니다. Next.js App Router에서 컴포넌트는 기본적으로 Server Component이기 때문에, `React.lazy`를 그대로 사용하면 서버 렌더링 단계에서 오류가 발생합니다.

`next/dynamic`은 `React.lazy`와 `Suspense`를 Next.js SSR 환경에 맞게 감싼 래퍼(wrapper)입니다.

| 구분              |                   `React.lazy`                    |           `next/dynamic`           |
| ----------------- | :-----------------------------------------------: | :--------------------------------: |
| SSR 지원          |                        ❌                         |                 ✅                 |
| `ssr: false` 옵션 |                        ❌                         |                 ✅                 |
| named export      | `.then(m => ({ default: m.Foo }))` 직접 감쌈 필요 | `.then(m => m.Foo)` 바로 사용 가능 |

```typescript
import dynamic from "next/dynamic";

const HeavyComponent = dynamic(() => import("./heavy-component").then((m) => m.HeavyComponent), { ssr: false });
```

> **📌 참고** `"use client"`를 선언해도 초기 HTML은 서버에서 렌더링됩니다. `ssr: false`는 한 단계 더 나아가 해당 컴포넌트를 초기 SSR HTML에서 완전히 제외합니다. `window`, `document` 같은 브라우저 전용 API를 사용하거나, SSR 환경(Node.js)에서 실행 자체가 불가능한 라이브러리를 사용하는 경우에 필요합니다.

### 청크 분리 타이밍

`next/dynamic`을 사용하면 Webpack이 빌드 시점에 해당 모듈을 별도 청크 파일로 분리합니다.

```
초기 로드: main.js (dynamic import 대상 컴포넌트 제외)
버튼 클릭 → 렌더링 요청 → [청크 fetch] → 파싱·평가 → 렌더링
```

초기 로드 시 해당 청크를 내려받지 않고, 컴포넌트가 실제로 렌더링되어야 하는 시점에 네트워크 요청이 발생합니다. 이 메커니즘으로 초기 JS 파싱·평가 비용이 분산되고 TBT(Total Blocking Time)가 개선됩니다.

## 2단계: 언제 적용해야 효과가 있는가

모든 컴포넌트에 `next/dynamic`을 적용한다고 성능이 좋아지지 않습니다. 아래 세 조건을 모두 충족해야 실질적인 효과가 있습니다.

```
조건 1. 이 컴포넌트가 조건부 렌더링 대상인가?
        (클릭, 폼 제출 등 사용자 행동 이후에만 마운트 — 페이지 접속 즉시 렌더링되지 않음)

조건 2. 이 컴포넌트만 독자적으로 사용하는 무거운 패키지가 있는가?
        (같은 페이지의 다른 컴포넌트가 이미 import하면 효과 없음)

조건 3. 이 컴포넌트가 현재 페이지의 LCP 요소가 아닌가?
        (LCP 요소라면 어떤 조건을 충족해도 lazy 처리 금지)
```

조건 1만 충족해도 청크는 분리되지만, 조건 2를 충족하지 않으면 패키지가 Webpack의 공통 청크에 어차피 포함됩니다.

### 조건 3: LCP 요소에 lazy 처리를 하면 안 되는 이유

LCP(Largest Contentful Paint)는 뷰포트에서 가장 큰 콘텐츠 요소가 렌더링되는 시점입니다. LCP 요소를 `next/dynamic`으로 분리하면, 해당 요소가 렌더링되기까지 청크 fetch 왕복이 추가됩니다.

```
[기존]
초기 번들 로드 → LCP 요소 파싱·평가 → 렌더링 → LCP 측정

[lazy 처리 후]
초기 번들 로드 → (청크 fetch) → 청크 도착 → 파싱·평가 → 렌더링 → LCP 측정
                      ↑
             이 구간만큼 LCP가 뒤로 밀림
```

초기 JS 평가 비용은 줄지만, LCP 측정 시점 자체가 늦어집니다. 조건 1, 2를 모두 충족하더라도 LCP 요소라면 적용 대상에서 제외해야 합니다.

> **⚠️ 주의** LCP 요소는 lazy 처리하지 않습니다. 페이지 진입 즉시 렌더링되어야 하는 요소를 지연 로딩하면 오히려 사용자 경험이 나빠집니다.

### 적용 예시

**`RegisterModal` — 공연 등록 폼**

```typescript
const RegisterModal = dynamic(() => import("./register-modal").then((m) => m.RegisterModal), { ssr: false });
```

- 조건 1: "공연 등록" 버튼 클릭 시에만 마운트
- 조건 2: `react-hook-form`을 이 컴포넌트만 독자적으로 사용
- 조건 3: LCP 요소 아님

**`ProfileEditModal` — 프로필 수정 폼**

```typescript
const ProfileEditModal = dynamic(() => import("./profile-edit-modal").then((m) => m.ProfileEditModal), { ssr: false });
```

- 조건 1: "수정하기" 버튼 클릭 시에만 마운트
- 조건 2: `/profile` 페이지에서 `react-hook-form`을 사용하는 컴포넌트가 이 모달뿐
- 조건 3: LCP 요소 아님

## 3단계: barrel export 경유하면 Code Splitting이 무효화된다

조건을 모두 충족하더라도, `index.ts`의 barrel export를 경유해서 dynamic import를 하면 Code Splitting이 작동하지 않습니다.

```typescript
// index.ts (barrel export)
export { RegisterModal } from "./register-modal";
export { OtherComponent } from "./other-component";
export { AnotherComponent } from "./another-component";
```

```typescript
// ❌ barrel export 경유 — Code Splitting 무효
const RegisterModal = dynamic(() => import("./index").then((m) => m.RegisterModal));
```

`index.ts`를 import하는 순간 Webpack은 해당 barrel 파일 전체를 하나의 모듈 그래프로 처리합니다. `OtherComponent`, `AnotherComponent`까지 동일 청크에 묶이고, 분리하려던 무거운 패키지도 함께 초기 번들에 포함됩니다.

```typescript
// ✅ 소스 파일 직접 import — Code Splitting 정상 작동
const RegisterModal = dynamic(() => import("./register-modal").then((m) => m.RegisterModal));
```

dynamic import의 경로는 반드시 **분리하려는 컴포넌트의 소스 파일을 직접** 가리켜야 합니다.

> **📌 참고** Next.js App Router에서는 각 Server Component 파일이 독립적인 진입점이 되므로 barrel export 문제가 상대적으로 적습니다. 그러나 Client Component 내부에서 `next/dynamic`을 사용할 때는 이 규칙이 그대로 적용됩니다.

## 4단계: `webpackChunkName`으로 lazy load 동작 확인하기

`next/dynamic`만 사용하면 청크 파일의 이름이 `chunks/4a2f9c.js`처럼 해시 기반으로 생성됩니다. Network 탭에서 어떤 컴포넌트의 청크인지 식별하기 어렵습니다.

### 청크 이름 지정

매직 주석 `/* webpackChunkName: "이름" */`으로 청크 이름을 명시적으로 지정할 수 있습니다.

**적용 전**

```typescript
const RegisterModal = dynamic(() => import("./register-modal").then((m) => m.RegisterModal), { ssr: false });
// → 생성 청크: chunks/4a2f9c.js (해시 기반)
```

**적용 후**

```typescript
const RegisterModal = dynamic(
  () =>
    import(
      /* webpackChunkName: "register-modal" */
      "./register-modal"
    ).then((m) => m.RegisterModal),
  { ssr: false },
);
// → 생성 청크: chunks/register-modal.js
```

### 확인 포인트

| 항목         | 확인 내용                                           |
| ------------ | --------------------------------------------------- |
| 초기 로드 시 | 해당 청크가 **요청되지 않는지** 확인                |
| 버튼 클릭 후 | 청크 파일이 새롭게 **요청·응답되는지** 확인         |
| 두 번째 클릭 | 캐시에서 응답되어 **재요청이 발생하지 않는지** 확인 |

> **📌 참고** 개발 환경(`next dev`)에서는 청크 이름이 프로덕션과 다르게 표시될 수 있습니다. 정확한 청크 이름 확인은 `next build && next start` 이후 진행하는 것을 권장합니다.

## 정리

`next/dynamic` 구현 자체는 단순합니다. 어려운 부분은 **이 컴포넌트에 적용해야 하는가**를 판단하는 것입니다.

- 사용자 행동 이후 조건부로 마운트되고, 독립적으로 사용하는 무거운 패키지가 있으며, LCP 요소가 아닌 컴포넌트에 적용합니다.
- barrel export를 경유하면 Webpack이 전체 모듈을 하나로 묶어 Code Splitting이 무효화됩니다.
- `webpackChunkName`으로 청크 이름을 지정하면 Network 탭에서 lazy load 동작을 명확히 검증할 수 있습니다.
