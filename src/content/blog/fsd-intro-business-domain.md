## 1. 들어가며

프론트엔드 프로젝트 규모가 커질수록 폴더 구조는 점점 복잡해집니다. `components/`, `hooks/`, `utils/` 같은 역할 기반 폴더는 처음엔 직관적으로 느껴지지만, 기능이 추가될수록 어느 파일이 어떤 화면을 위한 것인지 파악하기 어려워집니다.

**Feature-Sliced Design(FSD)** 은 이 문제를 해결하기 위해 등장한 프론트엔드 아키텍처 방법론입니다. FSD는 코드를 **비즈니스 도메인** 단위로 분리하여, 프로젝트를 처음 접하는 개발자도 코드의 목적을 쉽게 파악할 수 있도록 설계합니다.

이 글에서는 FSD가 무엇인지, 왜 비즈니스 도메인 중심 구조가 필요한지, 그리고 실제로 어떻게 구성하는지를 살펴봅니다.

---

## 2. 기존 역할 기반 구조의 문제점

역할 기반 폴더 구조는 파일의 **기술적 성격**에 따라 분류합니다.

```
src/
├── components/
│   ├── Button.tsx
│   ├── Modal.tsx
│   ├── UserCard.tsx
│   └── CartItem.tsx
├── hooks/
│   ├── useAuth.ts
│   ├── useCart.ts
│   └── useUser.ts
└── services/
    ├── authService.ts
    └── cartService.ts
```

이 구조의 문제는 다음과 같습니다.

- `components/` 폴더에 100개의 파일이 쌓이면 무엇을 위한 컴포넌트인지 파악하기 어렵습니다.
- 장바구니 기능을 수정하려면 `components/CartItem.tsx`, `hooks/useCart.ts`, `services/cartService.ts`를 각각 찾아야 합니다.
- 신규 팀원이 "장바구니 기능이 어디에 있나요?"라고 물었을 때 명확하게 답변하기 어렵습니다.

---

## 3. FSD: 비즈니스 도메인으로 분할하기

FSD는 코드를 **비즈니스 도메인** 단위로 나눕니다. e-commerce 애플리케이션을 예로 들면 다음과 같습니다.

```
src/
├── features/
│   ├── auth/        ← 사용자 로그인·인증
│   ├── cart/        ← 장바구니 담기·수정·삭제
│   ├── product/     ← 상품 조회·필터링
│   └── order/       ← 주문 생성·결제
├── entities/
│   ├── user/        ← 사용자 도메인 모델
│   └── product/     ← 상품 도메인 모델
└── shared/          ← 전역 공통 컴포넌트·유틸
```

이제 장바구니 기능과 관련된 모든 코드는 `features/cart/` 아래에 모여 있습니다. "장바구니 기능이 어디 있나요?"라는 질문에 `features/cart/`라고 명확히 답할 수 있습니다.

---

## 4. 비즈니스 언어로 모듈 이름 짓기

FSD는 모듈 이름을 **비즈니스 용어(domain language)** 로 짓도록 권장합니다.

| 기술적 이름 (피해야 할 방식) | 비즈니스 이름 (권장 방식) |
| ---------------------------- | ------------------------- |
| `CartComponent`              | `features/cart`           |
| `UserHandler`                | `features/auth`           |
| `ProductListModal`           | `features/product`        |

`features/cart`라는 이름만 보아도 이 모듈이 **고객이 상품을 담는 행위**와 관련된 코드임을 비즈니스 관점에서 즉시 이해할 수 있습니다. `CartComponent`처럼 기술적 용어로 지었다면 "어떤 역할을 하는 컴포넌트인가?" 추가로 파악해야 합니다.

---

## 5. 도메인 격리로 얻는 이점

비즈니스 도메인으로 코드를 분리하면 다음과 같은 이점이 생깁니다.

### 이해하기 쉬움

`features/auth`를 보면 로그인·인증 관련 코드임을 즉시 파악할 수 있습니다. 프로젝트 전체를 파악하지 않아도 해당 도메인에 집중해 개발할 수 있습니다.

### 독립성 유지

각 도메인이 독립적인 모듈로 분리되어 있어, 장바구니 기능 변경이 주문 기능에 영향을 주지 않습니다. 팀 내 역할 분담도 명확해집니다.

### 유지보수성 향상

기능 변경 시 수정해야 할 범위가 해당 도메인 폴더로 한정됩니다. 예상치 못한 사이드 이펙트가 줄어듭니다.

### 개발 효율성 향상

새로운 기능을 추가할 때 어디에 코드를 작성해야 할지 명확합니다. 비즈니스 요구사항을 구조에 그대로 반영할 수 있습니다.

---

## 6. 정리

FSD의 **비즈니스 도메인 지향성**은 단순한 폴더 정리 방식이 아닙니다. 코드 구조가 비즈니스 요구사항을 직접 반영하게 함으로써, 개발자와 비즈니스 담당자가 같은 언어로 대화할 수 있는 환경을 만듭니다.

다음 글에서는 FSD의 레이어 구조와 의존성 규칙에 대해 살펴보겠습니다.
