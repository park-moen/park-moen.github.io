> Redux Toolkit + Redux Saga + 내부 유틸리티 `reduxMaker`로 구성된 레거시 상태 관리 구조는 보일러플레이트를 줄였지만, 비표준 API와 제너레이터 문법으로 인해 신규 팀원 온보딩 러닝커브가 높고 서버 상태 캐싱을 직접 관리해야 했습니다. 서버 상태·클라이언트 상태·폼 상태를 각각 TanStack Query, Zustand, React Hook Form + Zod로 분리해 표준 생태계로 전환하고 `reduxMaker` 의존성을 완전 제거한 과정을 기록합니다.

---

## 1. 레거시 구조와 reduxMaker의 역할

기존 코드베이스는 Redux Toolkit + Redux Saga 조합이었습니다. 비동기 액션을 하나 추가할 때마다 Slice, Saga, 로딩/에러 상태를 각각 별도로 작성해야 했고, 이 반복을 줄이기 위해 내부 유틸리티 `reduxMaker`를 도입하였습니다.

### reduxMaker 구조

`reduxMaker`는 `prefix`(Slice 이름)와 비동기 요청 목록(`asyncRequests`)을 받아 Slice와 Saga를 한 번에 생성합니다.

```typescript
// src/features/auth/authReducer.ts
const asyncRequests = [
  { action: 'kakaoLogin', state: 'kakaoLoginData', initialState: null, api: kakaoLogin },
  { action: 'googleLogin', state: 'googleLoginData', initialState: null, api: googleLogin },
  { action: 'refreshToken', state: 'tokenData', initialState: null, api: refreshToken },
] as const

// reduxMaker 한 번 호출로 Slice·Saga·loading/error 상태가 자동 생성됩니다.
const module = reduxMaker(prefix, asyncRequests, localState, localReducers)
export const { slice: authSlice, actions: authAction, saga: authSaga } = module
```

→ [전체 코드 보기 (GitHub)](https://github.com/eco-dessert-platform/dessert-front-dashboard/blob/530460bff7f23bc12da2c0d433b3f0ee902dbb89/src/features/auth/authReducer.ts)

내부적으로 `reduxMaker`는 각 비동기 요청에 대해 `loading`, `success`, `fail` 액션을 자동 생성하고, `reducerUtils`로 상태를 관리합니다.

```typescript
// reduxUtils.ts — reducerUtils
export const reducerUtils = {
  init: <DataType>(defaultType: DataType | null = null): AsyncState<DataType> => ({
    data: defaultType,
    loading: false,
    error: false,
    errorMsg: "",
  }),

  loading: <DataType>(prevData: DataType | null = null): AsyncState<DataType> => ({
    data: prevData,
    loading: true,
    error: false,
    errorMsg: "",
  }),

  success: <DataType>(data: DataType): AsyncState<DataType> => ({
    data: data,
    loading: false,
    error: false,
    errorMsg: "",
  }),

  error: <DataType>(prevData: DataType | null = null, errorMsg: string): AsyncState<DataType> => ({
    data: prevData,
    loading: false,
    error: true,
    errorMsg: errorMsg,
  }),
};
```

비동기 흐름은 `createRequestSaga`가 담당합니다.

```typescript
// reduxUtils.ts — createRequestSaga
const createRequestSaga = <PayloadType, ResponseType>(
  prefix: string,
  reducerName: string,
  api: (payload: PayloadType) => Promise<AxiosResponse<ResponseType>>,
) => {
  return function* fetchApiData(action: AnyAction) {
    try {
      const response: AxiosResponse<ResponseType> = yield call(api, action.payload);
      yield put({ type: `${prefix}/${reducerName}Success`, payload: response.data });
    } catch (error: unknown) {
      yield put({ type: `${prefix}/${reducerName}Fail`, payload: extractErrorMessage(error, "...") });
    }
  };
};
```

→ [전체 코드 보기 (GitHub)](https://github.com/eco-dessert-platform/dessert-front-dashboard/blob/530460bff7f23bc12da2c0d433b3f0ee902dbb89/src/global/store/redux/reduxUtils.ts)

## 2. 문제 분석: reduxMaker의 한계

`reduxMaker`는 반복 코드를 줄이는 목적은 달성했지만, 세 가지 구조적 한계가 남았습니다.

### 문제 1: 비표준 API — 공식 문서로 학습 불가

`reduxMaker`는 이 프로젝트 내부에서만 통용되는 API입니다. 신규 팀원이 Redux Toolkit 공식 문서를 학습해도 `reduxMaker`의 인수 순서, 자동 생성되는 액션 이름(`${prefix}/${action}Success` 등)을 파악하려면 내부 구현 코드를 직접 분석해야 했습니다.

### 문제 2: 제너레이터 문법 — Redux Saga 학습 비용

Redux Saga는 `function*`과 `yield` 기반의 제너레이터 패턴을 사용합니다. `call`, `put`, `takeLatest` 같은 이펙트 함수는 Saga 고유 개념으로 별도 학습이 필요하며, 비동기 흐름 테스트 시 제너레이터 이터레이터를 직접 제어해야 하는 진입 장벽이 있었습니다.

### 문제 3: 서버 상태 캐싱 부재

`reducerUtils`로 관리하는 상태는 전역 Redux 스토어에 저장됩니다. 동일한 API를 여러 컴포넌트에서 사용하더라도 캐싱 전략이 없어 매 렌더링마다 네트워크 요청이 발생했습니다. `staleTime`, `refetchOnWindowFocus` 같은 서버 상태 최적화 기능도 부재했습니다.

## 3. 전환 전략: 상태 성격에 따른 라이브러리 분리

단일 라이브러리로 모든 상태를 처리하는 대신, 상태의 성격에 따라 역할을 분리하는 방향을 채택했습니다.

| 상태 종류                  | 기존                          | 변경 후                   |
| -------------------------- | ----------------------------- | ------------------------- |
| 서버 상태 (비동기 API)     | Redux + Saga + reduxMaker     | **TanStack Query**        |
| 클라이언트 상태 (UI, 전역) | Redux Slice                   | **Zustand**               |
| 폼 상태                    | Redux Slice + 수동 validation | **React Hook Form + Zod** |

세 라이브러리 모두 React 생태계의 표준으로 자리잡혀 있습니다. 공식 문서만으로 학습이 가능하며, 외부 경험을 보유한 팀원도 별도 내부 학습 없이 바로 프로젝트를 진행할 수 있습니다.

## 4. TanStack Query 도입: 서버 상태를 캐시로 관리

### queryOptions — 타입 안전성 확보

쿼리 옵션 객체를 일반 상수로 작성하면 `stallTime`처럼 오타가 발생해도 TypeScript가 에러를 감지하지 못합니다. `queryOptions()`로 감싸면 객체 리터럴 타입 검사가 적용되어 오타를 즉시 컴파일 단계에서 차단합니다.

```typescript
// ❌ 일반 상수 — 오타를 잡을 수 없음
const ordersQuery = {
  queryKey: ["order", "list"],
  queryFn: () => getOrders(),
  stallTime: 5000, // staleTime의 오타 — 타입 에러가 발생하지 않음
};

// ✅ queryOptions — 오타 즉시 감지
const ordersQuery = queryOptions({
  queryKey: ["order", "list"],
  queryFn: () => getOrders(),
  stallTime: 5000, // ❗ Type error: Object literal may only specify known properties
});
```

### 계층적 queryKey — 무효화 범위 제어

쿼리 키를 계층 구조로 정의하면 `invalidateQueries`의 무효화 범위를 정밀하게 제어할 수 있습니다.

```typescript
// entity/product/upload-approval/upload-approval.query.ts
import { queryOptions } from "@tanstack/react-query";
import { getUploadApprovals } from "./upload-approval.api";
import type { GetUploadApprovalsRequestParams } from "./upload-approval.type";

export const productQueries = {
  all: () => ["products"] as const,
  uploadApprovals: () => [...productQueries.all(), "upload-approvals"] as const,

  uploadApprovalList: (params: GetUploadApprovalsRequestParams = {}) =>
    queryOptions({
      queryKey: [...productQueries.uploadApprovals(), params],
      queryFn: () => getUploadApprovals(params),
    }),
};
```

→ [전체 코드 보기 (GitHub)](https://github.com/eco-dessert-platform/dessert-front-dashboard/blob/develop/apps/admin/src/entity/product/upload-approval/upload-approval.query.ts)

`productQueries.uploadApprovals()`를 무효화하면 `uploadApprovalList`의 모든 파라미터 조합이 일괄 무효화됩니다. Saga에서 수동으로 상태를 초기화하던 방식에 비해 선언적이며 예측 가능합니다.

### useMutation — 변경 후 자동 재조회

```typescript
// features/product/upload-approval/upload-approval.mutation.ts
import { toast } from "@dessert/ui";
import { useMutation, useQueryClient } from "@tanstack/react-query";
import { decideUploadApproval, productQueries } from "@/entity/product/upload-approval";
import type { DecideUploadApproval } from "@/entity/product/upload-approval";

export const useDecideUploadApprovalMutation = () => {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: ({ boardId, body }: { boardId: number; body: DecideUploadApproval }) =>
      decideUploadApproval(boardId, body),
    onSuccess: (_, { body }) => {
      queryClient.invalidateQueries({
        queryKey: productQueries.uploadApprovals(),
      });

      const message = body.decisionType === "APPROVE" ? "업로드 상품을 승인했습니다." : "업로드 상품을 거절했습니다.";
      toast.success(message);
    },
    onError: (_, { body }) => {
      const action = body.decisionType === "APPROVE" ? "승인" : "거절";
      toast.error(`업로드 상품 ${action}에 실패했습니다.`, "다시 시도해주세요.");
    },
  });
};
```

→ [전체 코드 보기 (GitHub)](https://github.com/eco-dessert-platform/dessert-front-dashboard/blob/develop/apps/admin/src/features/product/upload-approval/upload-approval.mutation.ts)

`onSuccess`에서 `invalidateQueries`를 호출하면 TanStack Query가 자동으로 최신 데이터를 재조회합니다. 레거시 구조에서 Saga가 처리하던 "변경 → 재조회" 흐름을 선언적으로 표현할 수 있습니다.

## 5. FSD 레이어 구조에서의 Query 배치 기준

TanStack Query를 FSD 레이어 구조와 함께 사용할 때의 배치 기준을 다음과 같이 정의했습니다.

```
entity/   ← queryOptions 정의, useQuery 호출 (읽기 전용)
features/ ← useMutation (데이터 변경)
```

| 구분                | 배치 레이어 | 이유                          |
| ------------------- | ----------- | ----------------------------- |
| `queryOptions` 정의 | `entity`    | 여러 features에서 재사용 가능 |
| `useQuery` 호출     | `pages`     | 페이지 단위로 데이터 구독     |
| `useMutation`       | `features`  | 특정 기능에만 귀속            |

`features` 레이어에는 `.api.ts`를 두지 않습니다. API 호출은 반드시 `entity`의 `query.ts`를 통해 이루어지며, 이 원칙 덕분에 수정 대상 파일을 파일명만으로 즉시 특정할 수 있습니다.

## 6. 전환 결과

### 온보딩 비용 절감

`reduxMaker` 내부 구현을 분석해야 했던 학습 경로가 제거됐습니다. TanStack Query, Zustand, React Hook Form, Zod 모두 공식 문서가 충분히 정비되어 있으며, 외부에서 해당 스택 경험을 보유한 채 합류하는 팀원도 추가 내부 학습 없이 프로젝트를 진행할 수 있습니다.

### 서버 상태 캐싱

`staleTime`, `gcTime`, `refetchOnWindowFocus` 등 TanStack Query의 캐싱 전략을 즉시 활용할 수 있게 됐습니다. 동일한 쿼리를 여러 컴포넌트에서 구독해도 네트워크 요청은 한 번만 발생합니다.

### 코드 추적 가능성

`reduxMaker`가 자동 생성하던 액션 이름(`${prefix}/${action}Success` 등)은 IDE의 Go to Definition이나 Find References로 추적하기 어려웠습니다. TanStack Query 전환 후에는 `queryKey`와 `mutationFn`이 명시적으로 작성되어 있어 API 호출 경로를 정적으로 추적할 수 있습니다.

### 비표준 의존성 제거

`reduxMaker`와 `reducerUtils`는 이 프로젝트 내부에서만 통용되는 유틸리티였습니다. 전환 완료 후 해당 파일을 코드베이스에서 완전히 삭제했습니다.
