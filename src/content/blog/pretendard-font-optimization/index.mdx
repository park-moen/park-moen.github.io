## 배경

UNIBUSK 홈 페이지의 Lighthouse Performance 점수가 63~70점으로 측정되었습니다.

실제 점수 저하의 주된 원인은 다음과 같았습니다.

| 항목                     | 영향        |
| ------------------------ | ----------- |
| Unused JavaScript        | 1,905 KiB   |
| HTTP/1.1 (HTTP/2 미적용) | ~810ms 지연 |
| Render-blocking CSS      | ~120ms 지연 |

그 중 Chrome DevTools Network 탭을 통해 `PretendardVariable.woff2` 파일이 2MB 크기로 26초에 걸쳐 로드되고 있음을 확인했습니다. `font-display: swap`이 적용되어 있어 폰트 로딩이 렌더링을 직접 차단하지는 않지만, 총 네트워크 페이로드(4,462 KiB 중 2,010 KiB)를 크게 차지하고 있었습니다. Lighthouse의 "Avoid enormous network payloads" 항목(Diagnostics)에서도 경고로 표시되었습니다.

  src={lighthouseBefore}

폰트 최적화만으로 Lighthouse 점수가 극적으로 오르지는 않지만, **실 사용자의 다운로드 데이터와 FOUT(Flash of Unstyled Text, 폰트 로딩 전 시스템 폰트로 먼저 렌더링되는 현상) 체감 시간을 줄이는 데 효과적입니다.** 이를 위해 3가지 방식을 실제로 적용하고 비교 측정했습니다.

## 3가지 방법 비교

### 방법 1: Variable Font 자체 호스팅 (기존 방식)

`next/font/local`을 사용해 `PretendardVariable.woff2` 단일 파일을 제공합니다.

```tsx
// app/layout.tsx
const pretendard = localFont({
  src: "../styles/fonts/PretendardVariable.woff2",
  display: "swap",
  weight: "100 900",
  variable: "--font-pretendard",
});
```

<br />

<Image src={fontVariable} alt="Variable Font 네트워크 측정 결과" width={720} />

| 항목      | 측정값       |
| --------- | ------------ |
| 파일 수   | 1개          |
| 크기      | **2,058 kB** |
| 로딩 시간 | **26.08s**   |
| Priority  | High         |

**성능 저하 원인**

Variable Font는 `font-weight: 100`부터 `900`까지 모든 굵기를 하나의 파일에 포함합니다. 파일 하나로 전체 굵기를 표현할 수 있다는 장점이 있으나, 그만큼 파일 크기가 커집니다. 한글은 초성·중성·종성 조합으로 최대 11,172자의 음절을 표현할 수 있기 때문에, 전체 글자셋을 포함한 Variable Font는 2MB에 달합니다.

---

### 방법 2: Subset 4개 자체 호스팅 (최종 선택)

굵기별로 분리된 정적 폰트 파일 4개를 `next/font/local`로 제공합니다.

```tsx
// app/layout.tsx
const pretendard = localFont({
  src: [
    { path: "../styles/fonts/Pretendard-Regular.subset.woff2", weight: "400" },
    { path: "../styles/fonts/Pretendard-Medium.subset.woff2", weight: "500" },
    { path: "../styles/fonts/Pretendard-SemiBold.subset.woff2", weight: "600" },
    { path: "../styles/fonts/Pretendard-Bold.subset.woff2", weight: "700" },
  ],
  display: "swap",
  variable: "--font-pretendard",
});
```

<br />

<Image src={fontSubset} alt="Subset 4개 네트워크 측정 결과" width={720} />

| 항목      | 측정값                             |
| --------- | ---------------------------------- |
| 파일 수   | 4개                                |
| 크기      | 각 267~271 kB (합산 **~1,076 kB**) |
| 로딩 시간 | **~8.8s**                          |
| Priority  | High                               |

각 파일이 특정 굵기 하나만 포함하므로 단일 파일 크기가 작습니다. 다만 한글은 초성·중성·종성 조합으로 최대 11,172자의 음절을 표현할 수 있기 때문에, 전체 글자셋을 포함한 파일 4개 합산 시 1MB를 초과합니다.

**HTTP/2 병렬 다운로드**

4개 파일이라고 해서 순차적으로 다운로드되지 않습니다. `next/font/local`은 빌드 타임에 4개 파일 모두에 대한 `<link rel="preload">` 태그를 HTML에 동시에 삽입합니다. HTTP/2 환경에서는 하나의 TCP 연결로 여러 요청을 동시에 주고받는 멀티플렉싱이 지원되어, 4개 파일이 병렬로 다운로드됩니다.

따라서 실제 대기 시간은 **4개 합산(~1,076 kB)이 아니라 가장 늦게 완료되는 파일 1개(~271 kB)** 기준입니다. Variable Font(2,058 kB) 단일 파일보다 총 전송량이 절반으로 줄고, 병렬 다운로드 효과까지 더해져 로딩 시간이 크게 단축됩니다.

**Variable Font와의 결정적 차이: weight 표현 범위**

|                    | Variable Font      | Static Font (Subset)                |
| ------------------ | ------------------ | ----------------------------------- |
| weight 지원 범위   | 100~900 연속값     | 선언한 값만 (400, 500, 600, 700)    |
| `font-weight: 350` | 지원               | 미지원 (가장 가까운 400으로 렌더링) |
| 중간값 보간        | 브라우저 자동 처리 | 미지원                              |

### 방법 3: CDN Dynamic Subset (시도 후 롤백)

jsDelivr CDN에서 제공하는 `pretendard-dynamic-subset.min.css`를 사용합니다.

```tsx
// app/layout.tsx
export default function RootLayout({ children }: { children: ReactNode }) {
  return (
    <html lang="ko">
      <head>
        <link rel="preconnect" href="https://cdn.jsdelivr.net" crossOrigin="anonymous" />
        <link
          rel="stylesheet"
          crossOrigin="anonymous"
          href="https://cdn.jsdelivr.net/gh/orioncactus/pretendard@v1.3.9/dist/web/static/pretendard-dynamic-subset.min.css"
        />
      </head>
      <body className="antialiased">{children}</body>
    </html>
  );
}
```

| 항목        | 측정값       |
| ----------- | ------------ |
| 파일 수     | 20개+        |
| 청크당 크기 | **10~20 kB** |
| 로딩 시간   | **~2.5s**    |
| Priority    | **Highest**  |

네트워크 전송량만 보면 가장 우수한 결과입니다. 그러나 **프로덕션 Lighthouse 검사에서 심각한 문제가 발견되었습니다.**

## CDN 방식의 실제 문제: Render-blocking

### 문제 확인

CDN 방식을 프로덕션에 배포한 후 Lighthouse를 측정한 결과, `pretendard-dynamic-subset.min.css`가 **Render-blocking 리소스**로 감지되었습니다.

<Image
  src={renderBlockingCdn}
  alt="CDN 방식 적용 시 Lighthouse Render-blocking 경고 — JSDelivr 930ms, 절감 예상 1,720ms"
  width={720}
/>

| 리소스                            | Transfer Size | Duration   |
| --------------------------------- | ------------- | ---------- |
| unibusk.site CSS (1st Party)      | 16.6 KiB      | 300 ms     |
| pretendard-dynamic-subset.min.css | **20.9 KiB**  | **930 ms** |

Lighthouse는 이 두 리소스로 인한 예상 절감 시간을 **1,720ms**로 산정했습니다.

### 왜 Render-blocking이 발생하는가

`<link rel="stylesheet">`는 브라우저가 해당 CSS를 완전히 다운로드하고 파싱할 때까지 페이지 렌더링을 차단합니다. 자체 서버에서 가져오는 1st Party CSS도 동일하지만, **외부 CDN에서 가져오는 경우 DNS 조회 → TCP 연결 → TLS 핸드셰이크 → 다운로드**의 전 과정이 차단 시간에 포함됩니다.

```
브라우저가 HTML 파싱 중 <link rel="stylesheet"> 발견
         ↓
DNS 조회 (cdn.jsdelivr.net)
         ↓
TCP + TLS 연결
         ↓
CSS 파일 다운로드 (20.9 KiB)
         ↓
CSS 파싱 완료
         ↓
렌더링 재개  ← 이 모든 과정이 차단 시간
```

Dynamic Subset은 다운로드 크기를 줄이는 데 효과적이지만, **CSS를 외부에서 동기적으로 로드하는 방식 자체를 바꾸지 않는 한 Render-blocking은 피할 수 없습니다.**

## next/font/local이 Render-blocking을 해결하는 원리

`next/font/local`은 폰트 파일을 빌드 타임에 처리하여, 런타임에 외부 CSS를 로드하지 않습니다.

**빌드 타임에 발생하는 일:**

1. `@font-face` 규칙을 생성하여 `<style>` 태그로 HTML에 직접 삽입
2. `<link rel="preload" as="font">` 태그를 자동 생성하여 폰트 파일을 최우선 fetch
3. 폰트 파일을 content-hash URL(`/static/media/Pretendard-Regular.abc123.woff2`)로 빌드하여 자동 캐싱

**런타임에 발생하는 일:**

- 브라우저는 이미 HTML에 포함된 `@font-face`를 즉시 파싱
- 외부 도메인 왕복 없이 자체 서버에서 폰트 파일만 fetch
- Render-blocking CSS 요청 없음

## public/fonts/ 대신 src 내부 경로를 사용하는 이유

`next/font/local`을 사용할 때 폰트 파일을 `public/fonts/`에 두는 경우가 있습니다. 그러나 이 방식은 Next.js의 폰트 최적화를 전혀 활용하지 못합니다.

| 항목                   | `src/styles/fonts/` (next/font/local) | `public/fonts/` (직접 참조) |
| ---------------------- | ------------------------------------- | --------------------------- |
| 빌드 타임 처리         | ✅ Next.js 파이프라인에서 처리        | ❌ 빌드 없이 그대로 서빙    |
| content-hash URL       | ✅ 자동 생성 (캐시 무효화 자동 처리)  | ❌ 수동 관리 필요           |
| `@font-face` 인라인    | ✅ HTML에 직접 삽입                   | ❌ 별도 CSS 파일 필요       |
| `rel="preload"` 자동화 | ✅ Next.js가 자동 삽입                | ❌ 직접 작성 필요           |

`public/` 폴더는 "빌드 없이 그대로 서빙"이 목적입니다. `next/font/local`이 Render-blocking을 해결하는 핵심 메커니즘은 **빌드 타임 처리**이므로, 폰트 파일은 소스 트리 내부(예: `src/styles/fonts/`)에 위치해야 합니다.

## 최종 결과 비교

| 방법                         | 파일 수 | 총 크기         | 로딩 시간 | Priority | Render-blocking  |
| ---------------------------- | ------- | --------------- | --------- | -------- | ---------------- |
| Variable Font (자체 호스팅)  | 1개     | 2,058 kB        | 26.08s    | High     | 없음             |
| CDN Dynamic Subset           | 20개+   | 10 ~ 20 kB/청크 | ~2.5s     | Highest  | **930ms (문제)** |
| **Subset 4개 (자체 호스팅)** | **4개** | **~1,076 kB**   | **~8.8s** | **High** | **없음**         |

네트워크 전송량 기준으로는 CDN Dynamic Subset이 가장 우수하지만, Render-blocking으로 인한 FCP/LCP 저하가 실 사용자 체감에 더 직접적인 영향을 미칩니다. `next/font/local` 기반 Subset 자체 호스팅은 Variable Font 대비 파일 크기를 약 절반으로 줄이면서 Render-blocking을 발생시키지 않는 균형 잡힌 방식입니다.
