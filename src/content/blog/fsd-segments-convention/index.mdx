> FSD 레이어 구조는 있었지만, Slice 내부를 어떻게 나눌지 기준이 없었습니다. 그 결과 같은 `features` 레이어 안에서도 슬라이스마다 구조가 달랐고, 새 파일을 추가할 때마다 "어디에 두지?"라는 논의가 반복됐습니다. 이 글은 파일 suffix 규칙과 팀 3원칙을 정리해 반복되는 배치 논의를 없앤 과정을 기록합니다.

---

## 1. 문제 상황: FSD는 있지만 Segments 기준이 없었다

프로젝트에는 이미 FSD 레이어 구조가 적용되어 있었습니다.

```
src/
├── app/
├── pages/
├── features/
├── entity/
└── shared/
```

당시 팀은 대부분 새로 합류한 상태였습니다. FSD 폴더 구조는 이미 존재했지만, 어떤 파일을 어느 레이어에 두어야 하는지, 슬라이스 내부를 어떻게 나누는지에 대한 기준이 문서화되어 있지 않았습니다. 결과적으로 팀원마다 구조를 다르게 해석해 사용하고 있었습니다.

FSD 공식 문서는 레이어 간 의존성 규칙은 엄격하게 명시하지만, 슬라이스 내부(**Segments**)를 어떤 단위로 나눌지는 팀 자율에 맡깁니다. 기준이 없으니 각자의 판단대로 구조를 잡게 됐고, 그 차이가 쌓이면서 혼란이 생겼습니다.

같은 `features` 레이어 안에 있는 두 슬라이스를 보겠습니다.

두 슬라이스는 같은 `features` 레이어에 속하지만 구조가 완전히 달랐습니다. 이 상태에서 새로운 파일을 추가하면 자연스럽게 "이 파일은 어디에 두어야 하나?"라는 논의가 회의에서 반복됐습니다.

## 2. 원인: Segments 레벨에 기준이 없었다

FSD는 레이어 간 규칙(단방향 의존성, Public API 등)은 명확하게 정의합니다. 하지만 슬라이스 내부 Segments 구조에 대해서는 구체적인 규칙을 강제하지 않습니다.

팀원 A는 역할별 폴더로 나눴고, 팀원 B는 파일 이름에 도메인명을 붙여 구분했습니다. 어느 쪽도 틀리지 않았지만, 기준이 통일되지 않으니 코드베이스 전체에 걸쳐 구조가 일관성 없이 달라졌습니다.

이 상황을 해소하려면 세 가지가 필요했습니다.

1. 슬라이스 내부 파일의 역할을 구분하는 **파일 이름 규칙**
2. 레이어 간 참조 방향을 통제하는 **의존성 원칙**
3. TanStack Query 코드의 레이어 배치를 결정하는 **배치 기준**

## 3. 해결 1: 파일 suffix 규칙 도입

서브디렉토리로 Segments를 나누는 방식은 슬라이스 규모가 작을 때 오버엔지니어링이 됩니다. `hooks/useOrderList.ts` 하나 때문에 폴더를 만들어야 하는 상황이 생깁니다.

대신 **파일 접미사(suffix)로 역할을 명시**하는 방식을 채택했습니다.

| 접미사         | 역할                               |
| -------------- | ---------------------------------- |
| `.api.ts`      | axios 호출 함수                    |
| `.query.ts`    | `queryKey` 팩토리 + `queryOptions` |
| `.mutation.ts` | `useMutation` 훅 (데이터 변경)     |
| `.contract.ts` | 서버 응답 구조 검증 Zod 스키마     |
| `.schema.ts`   | 폼·입력값 검증 Zod 스키마          |
| `.type.ts`     | 도메인 타입 · 인터페이스           |
| `.hook.ts`     | UI 상태 관리 훅                    |
| `.ui.tsx`      | 순수 프레젠테이션 컴포넌트         |
| `.constant.ts` | 상태 라벨, 색상, UI 설정 상수      |
| `.mock.ts`     | 개발·테스트용 목 데이터            |

**entity/product/upload-approval 슬라이스 적용 예시:**

```
entity/product/upload-approval/
├── index.ts
├── upload-approval.api.ts
├── upload-approval.contract.ts
├── upload-approval.mock.ts
├── upload-approval.query.ts
└── upload-approval.type.ts
```

파일 목록만 봐도 어떤 파일이 어떤 역할인지 바로 알 수 있습니다. "API 함수를 수정해야 한다"면 `upload-approval.api.ts`를, "타입이 틀렸다"면 `upload-approval.type.ts`를 수정하면 됩니다.

## 4. 해결 2: 팀 3원칙 정리

파일 이름 규칙만으로는 레이어 간 참조 방향에 대한 논의를 막을 수 없었습니다. 레이어 구조가 의미를 갖기 위해 세 가지 원칙을 팀 가이드로 정리했습니다.

### 규칙 1: 단방향 의존성

```
app → pages → features → entity → shared
```

각 레이어는 자신보다 하위 레이어만 import할 수 있습니다.

```typescript
// ✅ features가 entity를 참조 — 허용
import { orderKeys } from "@/entity/order";

// ❌ entity가 features를 참조 — 위반
import { useOrderFilter } from "@/features/order";

// ❌ features/order가 features/product를 참조 — 위반
import { ProductSelector } from "@/features/product";
```

이 규칙을 어기면 한 모듈을 수정할 때 예측할 수 없는 곳이 깨지고, 순환 참조로 빌드가 실패할 가능성이 생깁니다.

### 규칙 2: Public API — `index.ts`를 통해서만 노출

슬라이스 내부 파일에 직접 접근하는 것을 금지합니다.

```typescript
// entity/order/index.ts — Public API 정의
export type { Order, OrderStatus, OrderFilters } from "./order.type";
export { orderQueries } from "./order.query";
export { ORDER_STATUS_LABELS, ORDER_ACTION_BAR_CONFIG } from "./order.constant";
```

```typescript
// ✅ index.ts를 통해 접근
import { Order, orderQueries } from "@/entity/order";

// ❌ 내부 파일에 직접 접근 — 위반
import { Order } from "@/entity/order/order.type";
import { orderQueries } from "@/entity/order/order.query";
```

내부 구현을 리팩토링해도 `index.ts` 바깥에는 영향이 없습니다. `index.ts`만 보면 슬라이스가 외부에 무엇을 제공하는지 한눈에 파악할 수 있습니다.

### 규칙 3: Cross-import 금지

같은 레이어에 속한 슬라이스끼리는 서로 참조할 수 없습니다.

```typescript
// ❌ features/order가 features/product를 참조 — Cross-import!
import { ProductSelector } from "@/features/product";
```

Cross-import가 발생하는 상황과 올바른 해결 방법은 다음과 같습니다.

| 상황                           | 잘못된 방법                                   | 올바른 방법                           |
| ------------------------------ | --------------------------------------------- | ------------------------------------- |
| 두 features가 같은 데이터 필요 | `features/a`에서 `features/b` import          | `entity/x`에 Query 정의 후 공유       |
| 두 features가 같은 UI 필요     | `features/a`에서 `features/b` 컴포넌트 import | `shared/ui` 또는 `entity/x/ui`로 이동 |
| 두 pages가 같은 로직 필요      | `pages/a`에서 `pages/b` import                | `features/x`로 로직 추출              |

## 5. 해결 3: TanStack Query 레이어 배치 기준

TanStack Query 코드를 어느 레이어에 두어야 할지는 팀 내 논의가 가장 많이 발생하는 지점이었습니다. 다음 기준으로 정리했습니다.

```
entity/   ← useQuery  (읽기 전용)
features/ ← useMutation (데이터 변경)
```

**entity에 useQuery 배치:**

```typescript
// entity/order/order.query.ts
import { queryOptions } from "@tanstack/react-query";
import { fetchOrderList } from "./order.api";

export const orderKeys = {
  all: ["orders"] as const,
  list: (params: OrderListParams) => [...orderKeys.all, "list", params] as const,
};

export const orderQueries = {
  list: (params: OrderListParams) =>
    queryOptions({
      queryKey: orderKeys.list(params),
      queryFn: () => fetchOrderList(params),
    }),
};
```

주문 목록은 주문 페이지, 대시보드, 통계 features 등 여러 곳에서 재사용됩니다. `entity`에 두면 중복 없이 공유할 수 있습니다.

**features에 useMutation 배치:**

```typescript
// features/order/order-action-bar/order-action-bar.mutation.ts
import { useMutation, useQueryClient } from "@tanstack/react-query";
import { orderKeys } from "@/entity/order";
import { confirmOrder } from "../order.api";

export const useConfirmOrder = () => {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: confirmOrder,
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: orderKeys.all });
    },
  });
};
```

주문 확인 mutation은 "주문 액션 바" 기능에만 사용됩니다. 해당 features에 두는 것이 응집도를 높입니다.

이 기준을 정리하면 다음과 같습니다.

| 훅 종류              | 배치 레이어 | 이유                     |
| -------------------- | ----------- | ------------------------ |
| `useQuery` (조회)    | `entity`    | 여러 features에서 재사용 |
| `useMutation` (변경) | `features`  | 특정 기능에만 귀속       |

## 6. 결과: 레이어 배치 논의의 구조적 제거

세 가지를 정리한 이후 달라진 것은 하나입니다. 파일을 어디에 둘지 물어볼 필요가 없어졌습니다.

- **파일을 추가할 때**: suffix를 보면 역할이 결정됩니다. `.api.ts`는 `entity`, `.hook.ts`와 `.ui.tsx`는 `features`.
- **레이어 간 import를 작성할 때**: 방향이 하위를 향하는지 확인하면 됩니다.
- **TanStack Query 훅을 추가할 때**: 읽기면 `entity`, 쓰기면 `features`.

PR 리뷰에서 "이 파일 어디에 두어야 하나요?" 대신 "이 비즈니스 로직이 맞나요?"를 논의할 수 있게 됐습니다. 구조에 대한 논의를 제거했더니 리뷰 시간이 실질적인 내용에 집중됐습니다.
