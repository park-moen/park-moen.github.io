> 웹 성능 지표의 정의를 정리하고, Lighthouse 점수와 실제 사용자 데이터 기반 성능 개선을 구분하는 방법을 설명합니다.

프론트엔드 성능을 분석하다 보면 FCP, LCP, CLS, TBT, TTFB, TTI, INP 같은 지표를 계속 만나게 됩니다. 이 지표들은 모두 "페이지가 빠른가?"를 하나의 숫자로 표현하지 않습니다. 각 지표는 네트워크 응답, 초기 렌더링, 주요 콘텐츠 렌더링, 시각적 안정성, 메인 스레드 점유, 사용자 입력 반응성처럼 서로 다른 구간을 측정합니다.

성능 측정에서 먼저 확인해야 할 질문은 다음과 같습니다.

> **이 지표가 어떤 사용자 시나리오와 어떤 병목 구간을 설명하는가?**

Lighthouse 점수는 초기 진단 지표로 사용할 수 있습니다. 하지만 Lighthouse는 정해진 환경에서 페이지를 로드해 생성한 **Lab 데이터(실험실 데이터)** 를 제공합니다. 실제 사용자는 다양한 기기, 네트워크, 브라우저 상태, 캐시 상태, 상호작용 패턴을 가지고 서비스를 사용합니다. 따라서 실무에서는 Lighthouse를 병목 후보를 찾는 도구로 사용하고, 실제 개선 여부는 **필드 데이터(Field Data)** 로 검증해야 합니다.

## 성능 지표는 무엇을 측정하는가

성능 지표는 페이지 로드와 상호작용 흐름을 시간순으로 나누어 보면 더 명확해집니다.

```text
요청 시작
  ↓
서버가 첫 바이트 응답
  ↓
첫 콘텐츠가 화면에 표시
  ↓
주요 콘텐츠가 화면에 표시
  ↓
사용자가 클릭, 입력, 탭
  ↓
브라우저가 반응하고 다음 화면을 그림
```

각 지표는 이 흐름의 서로 다른 지점을 측정합니다.

| 지표 | 전체 이름                 | 측정 대상                                     |
| ---- | ------------------------- | --------------------------------------------- |
| TTFB | Time to First Byte        | 문서 요청 후 첫 응답 바이트를 받기까지의 시간 |
| FCP  | First Contentful Paint    | 첫 콘텐츠가 렌더링되기까지의 시간             |
| LCP  | Largest Contentful Paint  | 뷰포트 내 최대 콘텐츠 요소가 렌더링되는 시간  |
| CLS  | Cumulative Layout Shift   | 예기치 않은 레이아웃 이동의 누적 점수         |
| TBT  | Total Blocking Time       | 초기 로드 중 메인 스레드가 차단된 시간        |
| TTI  | Time to Interactive       | 안정적인 상호작용 가능 상태에 도달하는 시간   |
| INP  | Interaction to Next Paint | 사용자 입력부터 다음 페인트까지의 지연 시간   |

## 주요 성능 지표 정리

### TTFB: 첫 응답까지의 시간

`TTFB(Time to First Byte)`는 브라우저가 문서 요청을 보낸 뒤 서버의 첫 번째 바이트를 받기까지 걸린 시간입니다.

TTFB는 화면 렌더링 자체를 직접 측정하지는 않습니다. 하지만 HTML 응답이 지연되면 FCP와 LCP의 시작 시점도 함께 뒤로 밀립니다. 따라서 TTFB는 렌더링 지표에 영향을 주는 **선행 네트워크/서버 응답 지표**로 볼 수 있습니다.

TTFB가 느릴 때는 보통 다음 영역을 의심합니다.

| 원인           | 예시                                        |
| -------------- | ------------------------------------------- |
| 서버 처리 지연 | SSR 데이터 조회, 인증 처리, 복잡한 API 조합 |
| 네트워크 지연  | 사용자와 서버 리전이 멀리 떨어진 경우       |
| 캐시 부재      | CDN, HTML 캐시, API 캐시 미적용             |
| 리다이렉트     | HTTP에서 HTTPS로 이동, 불필요한 라우팅      |

> **참고**
> TTFB가 낮아도 전체 로딩 성능이 자동으로 개선되는 것은 아닙니다. HTML 응답 이후 JavaScript 실행, CSS 로딩, 이미지 로딩이 지연되면 FCP와 LCP는 여전히 악화될 수 있습니다.

### FCP: 첫 콘텐츠가 보이는 시간

`FCP(First Contentful Paint)`는 사용자가 화면에서 첫 번째 콘텐츠를 볼 수 있는 시점을 측정합니다. 여기서 콘텐츠는 텍스트, 이미지, SVG, canvas 등 실제로 의미 있는 화면 요소를 말합니다.

FCP는 초기 빈 화면 상태가 해소되는 시점을 판단할 때 사용할 수 있습니다.

```text
FCP 지연 예:
페이지 접속 → 흰 화면 3초 → 콘텐츠 등장

FCP 개선 예:
페이지 접속 → 0.8초 뒤 헤더나 텍스트 등장 → 나머지 콘텐츠 순차 로딩
```

FCP를 개선하려면 초기 HTML, CSS, 폰트, 렌더링 차단 리소스를 봐야 합니다. 다만 FCP는 "첫 콘텐츠"만 측정합니다. 페이지의 주요 콘텐츠가 늦게 렌더링되는 경우에는 FCP만으로 로딩 성능을 판단하기 어렵습니다.

### LCP: 주요 콘텐츠가 보이는 시간

`LCP(Largest Contentful Paint)`는 뷰포트 안에서 가장 큰 이미지 또는 텍스트 블록이 렌더링된 시점을 측정합니다. Core Web Vitals의 안정 지표 중 하나이며, 주요 콘텐츠 렌더링 완료 시점을 추정하는 데 사용됩니다.

예를 들어 랜딩 페이지라면 히어로 이미지, 상품 상세 페이지라면 상품 이미지나 상품 제목 영역, 블로그라면 제목과 본문 첫 영역이 LCP 후보가 될 수 있습니다.

LCP 개선에서는 먼저 실제 LCP 요소를 확인해야 합니다.

1. Chrome DevTools Performance 패널에서 LCP 마커를 확인합니다.
2. Lighthouse 결과의 Largest Contentful Paint element를 확인합니다.
3. 이미지라면 크기, 포맷, preload, fetch priority, CDN 상태를 확인합니다.
4. 텍스트라면 폰트 로딩, CSS 렌더링 차단, 서버 응답 지연을 확인합니다.

> **주의**
> 개발자가 중요하다고 생각한 요소와 브라우저가 LCP로 판정한 요소가 다를 수 있습니다. LCP는 "의미상 중요한 요소"가 아니라 "뷰포트 내 가장 큰 콘텐츠 요소"를 기준으로 계산됩니다.

### CLS: 레이아웃 이동 누적 점수

`CLS(Cumulative Layout Shift)`는 페이지 생명주기 동안 예기치 않게 발생한 레이아웃 이동의 누적 점수입니다. 이미지, 광고, 배너, 폰트, 동적으로 삽입되는 UI가 기존 콘텐츠의 위치를 밀어내면 CLS가 증가합니다.

CLS는 시간 단위 지표가 아닙니다. LCP나 INP처럼 "몇 ms가 걸렸는가"를 측정하지 않고, 화면에 이미 렌더링된 요소의 위치가 얼마나 불안정하게 바뀌었는지를 점수로 계산합니다.

CLS가 높을 때는 다음 원인을 확인합니다.

| 원인              | 예시                                                       |
| ----------------- | ---------------------------------------------------------- |
| 이미지 크기 미지정 | `width`, `height`, `aspect-ratio` 누락                     |
| 영역 예약 누락    | 광고, 배너, 추천 영역이 콘텐츠 로드 후 뒤늦게 삽입됨       |
| 폰트 교체         | 웹폰트 로딩 후 텍스트 크기, 줄바꿈, 줄 높이가 변경됨       |
| 동적 UI 삽입      | 상단 알림, 토스트, lazy component가 기존 콘텐츠를 밀어냄   |

CLS 개선에서는 요소가 렌더링되기 전에 필요한 공간을 미리 확보하는 것이 중요합니다. 이미지에는 명시적인 크기나 `aspect-ratio`를 지정하고, 광고나 배너처럼 나중에 로드되는 영역은 placeholder나 최소 높이를 통해 레이아웃 공간을 예약해야 합니다.

### TBT: 메인 스레드가 막힌 총 시간

`TBT(Total Blocking Time)`는 FCP 이후부터 TTI까지, 메인 스레드가 긴 작업(Long Task)으로 인해 사용자 입력에 즉시 반응하지 못한 시간을 합산한 지표입니다.

브라우저의 메인 스레드는 JavaScript 실행, 스타일 계산, 레이아웃, 페인트 작업을 처리합니다. 이 스레드가 긴 JavaScript 작업으로 오래 점유되면 사용자가 버튼을 눌러도 브라우저가 바로 반응하지 못합니다.

Lighthouse에서 TBT는 특히 중요합니다. Lighthouse Performance 점수에서 높은 가중치를 가지며, 초기 로드 중 JavaScript 실행 비용을 파악하는 데 사용할 수 있습니다.

```text
70ms 작업이 실행됨
  └─ 50ms까지는 허용
  └─ 초과한 20ms가 blocking time으로 계산됨
```

TBT가 높을 때는 다음 개선이 효과적입니다.

| 개선 방향              | 예시                                                    |
| ---------------------- | ------------------------------------------------------- |
| JavaScript 줄이기      | 불필요한 패키지 제거, tree shaking 확인                 |
| 코드 분할              | 초기 렌더링에 필요 없는 모달, 차트, 에디터 lazy loading |
| 긴 작업 쪼개기         | 큰 계산을 작은 작업으로 분리                            |
| 서드파티 스크립트 관리 | 광고, 분석, 채팅 위젯 로딩 시점 조정                    |

### TTI: 페이지가 상호작용 가능해지는 시간

`TTI(Time to Interactive)`는 페이지가 시각적으로 표시되었을 뿐 아니라, 사용자 입력에 안정적으로 반응할 수 있는 상태가 되는 시점을 측정합니다.

하지만 최근에는 TTI의 중요도가 예전보다 낮아졌습니다. 이유는 TTI가 단일 로드 시점 중심 지표라 실제 사용자 상호작용 전체를 잘 설명하지 못하기 때문입니다. Chrome 문서에서도 LCP, TBT, INP 같은 최신 지표가 TTI보다 더 나은 대안으로 언급됩니다.

실무에서는 TTI를 주요 목표 지표로 삼기보다, 다음처럼 보조 지표로 해석하는 편이 적절합니다.

| TTI를 보는 상황          | 해석                                                           |
| ------------------------ | -------------------------------------------------------------- |
| Lighthouse 결과에서 참고 | 초기 로딩 중 상호작용 가능 상태가 늦어지는지 확인              |
| TBT와 함께 확인          | JavaScript 실행 비용 때문에 interactive 상태가 늦어지는지 추정 |
| 실제 UX 개선 목표        | INP와 사용자 플로우 기반 측정으로 대체                         |

### INP: 입력부터 다음 화면 반응까지의 시간

`INP(Interaction to Next Paint)`는 사용자의 클릭, 탭, 키보드 입력 같은 상호작용이 발생한 뒤 브라우저가 다음 프레임을 그리기까지 걸린 시간을 측정합니다.

INP는 2024년부터 Core Web Vitals의 안정 지표가 되었고, 기존의 FID(First Input Delay)를 대체했습니다. FID가 첫 입력의 지연만 측정했다면, INP는 페이지 생명주기 전체에서 발생하는 상호작용을 관찰합니다. 따라서 런타임 반응성(responsiveness)을 더 넓은 범위에서 설명합니다.

INP는 크게 세 구간으로 나눌 수 있습니다.

```text
사용자 입력
  ↓
입력 지연(input delay)
  ↓
이벤트 핸들러 실행(processing time)
  ↓
다음 화면 렌더링(presentation delay)
  ↓
다음 페인트 완료
```

INP가 높다는 것은 단순히 "클릭 이벤트 핸들러가 느리다"는 뜻만은 아닙니다. 입력 직전에 메인 스레드가 다른 작업으로 점유되어 있었을 수도 있고, 이벤트 핸들러 이후 렌더링 비용이 컸을 수도 있습니다.

## Lighthouse는 무엇을 잘하고, 무엇을 못하는가

Lighthouse는 성능 개선의 초기 진단 도구로 적합합니다. 재현 가능한 조건에서 페이지를 측정하고, 개선 기회를 Diagnostics와 Opportunities로 보여주기 때문입니다.

하지만 Lighthouse 결과만으로 실제 사용자 경험을 결론 내리기는 어렵습니다.

| 구분        | Lighthouse 중심 측정                         | 실제 사용자 경험 중심 측정                                |
| ----------- | -------------------------------------------- | --------------------------------------------------------- |
| 데이터 종류 | Lab 데이터                                   | 필드 데이터                                               |
| 측정 환경   | 통제된 기기, 네트워크, 브라우저 조건         | 실제 사용자의 기기, 네트워크, 브라우저                    |
| 측정 시점   | 보통 페이지 로드 1회                         | 실제 방문과 상호작용 전체                                 |
| 장점        | 재현 가능, 원인 분석 용이, 개발 중 확인 가능 | 실제 사용자 영향 확인 가능                                |
| 한계        | 사용자 다양성 반영 부족, INP 측정 제한       | 원인 분석에는 추가 계측 필요                              |
| 대표 도구   | Lighthouse, Chrome DevTools, WebPageTest     | CrUX, PageSpeed Insights 필드 데이터, Search Console, RUM |

Lighthouse가 특히 잘하는 일은 다음과 같습니다.

1. 초기 로딩 병목을 빠르게 찾습니다.
2. LCP 후보 요소와 렌더링 차단 리소스를 찾습니다.
3. TBT를 통해 JavaScript 실행 비용을 확인합니다.
4. 같은 조건에서 변경 전후를 비교합니다.

반대로 Lighthouse가 부족한 부분도 명확합니다.

1. 실제 사용자의 다양한 기기 성능을 대표하지 못합니다.
2. 사용자가 어떤 버튼을 언제 누르는지 알 수 없습니다.
3. 페이지 체류 중 발생하는 느린 상호작용을 충분히 포착하지 못합니다.
4. Lighthouse의 TBT는 INP의 대리 지표(proxy)일 수 있지만, INP 자체를 대체하지는 못합니다.

## Lighthouse 점수 개선과 사용자 경험 개선의 차이

Lighthouse 점수를 개선하는 작업과 실제 사용자 경험을 개선하는 작업은 겹치는 부분이 많지만 동일하지 않습니다.

예를 들어 Lighthouse에서 TBT가 높게 나와 JavaScript 번들을 줄였다고 가정해 보겠습니다. 이 작업은 초기 로드 성능에는 도움이 될 수 있습니다. 하지만 주요 병목이 "상품 필터 클릭 후 결과 목록 갱신이 지연되는 것"이라면, 초기 번들 크기만 줄여서는 해당 런타임 병목이 해결되지 않을 수 있습니다.

성능 개선의 목표는 점수 자체가 아니라 사용자 시나리오의 지연 시간을 줄이는 것입니다.

| 목표          | 잘못된 질문                       | 더 나은 질문                           |
| ------------- | --------------------------------- | -------------------------------------- |
| 로딩 개선     | Lighthouse 100점을 만들 수 있는가 | LCP 요소가 더 빠르게 렌더링되는가      |
| 반응성 개선   | TBT를 낮췄는가                    | 주요 상호작용의 INP가 개선되는가       |
| 안정성 개선   | 한 번 측정한 점수가 개선되었는가  | p75 기준에서도 개선이 유지되는가       |
| 우선순위 결정 | 점수 감점이 큰 항목부터 할까      | 트래픽과 영향도가 큰 병목부터 해결할까 |

## 실제 사용자 경험 개선을 위한 측정 방법

### 1. Lab 데이터와 필드 데이터를 역할별로 나눕니다

성능 측정은 하나의 도구로 끝내기보다 목적에 따라 나누어야 합니다.

| 단계         | 목적                               | 추천 데이터                                          |
| ------------ | ---------------------------------- | ---------------------------------------------------- |
| 문제 발견    | 어떤 지표가 나쁜지 파악            | CrUX, PageSpeed Insights 필드 데이터, Search Console |
| 원인 추정    | 어떤 리소스나 코드가 문제인지 확인 | Lighthouse, DevTools Performance                     |
| 재현         | 특정 페이지와 플로우에서 병목 재현 | DevTools, WebPageTest, 로컬 production 빌드          |
| 배포 후 검증 | 실제 사용자에게 개선됐는지 확인    | RUM, p75 Core Web Vitals                             |

권장 흐름은 실제 사용자 데이터로 문제를 발견하고, Lab 도구로 원인을 분석한 뒤, 다시 실제 사용자 데이터로 개선 여부를 확인하는 것입니다.

### 2. 평균이 아니라 p75를 봅니다

Core Web Vitals에서는 일반적으로 75번째 백분위수(p75)를 기준으로 봅니다. p75는 전체 방문 중 75%가 이 값 이하의 경험을 했다는 뜻입니다.

평균만 보면 느린 사용자가 가려질 수 있습니다.

```text
사용자 A: LCP 1.2s
사용자 B: LCP 1.4s
사용자 C: LCP 1.5s
사용자 D: LCP 6.0s

평균: 2.0s
```

평균만 보면 문제가 작아 보일 수 있지만, 느린 네트워크나 저사양 기기 사용자의 지연 시간은 여전히 높을 수 있습니다. 성능 개선에서는 평균뿐 아니라 p75, p90 같은 상위 지연 구간을 함께 봐야 합니다.

### 3. 페이지 단위가 아니라 사용자 플로우 단위로 봅니다

Lighthouse는 특정 URL의 초기 로드를 잘 측정합니다. 하지만 실제 사용자는 페이지에 들어온 뒤 검색하고, 필터를 바꾸고, 모달을 열고, 폼을 제출합니다.

따라서 실제 서비스에서는 다음처럼 사용자 플로우 기준으로 측정 범위를 정의해야 합니다.

| 플로우         | 봐야 할 지표                           |
| -------------- | -------------------------------------- |
| 홈 진입        | TTFB, FCP, LCP                         |
| 상품 목록 진입 | LCP, API 응답 시간, 이미지 로딩        |
| 필터 변경      | INP, API 응답 시간, 렌더링 비용        |
| 검색 입력      | INP, debounce 동작, 리스트 렌더링 비용 |
| 결제 버튼 클릭 | INP, mutation 지연, loading state 표시 |

이렇게 분리하면 "페이지 로딩 성능"과 "사용 중 성능"을 다른 지표와 다른 원인으로 분석할 수 있습니다.

### 4. RUM으로 실제 상호작용을 계측합니다

`RUM(Real User Monitoring)`은 실제 사용자의 브라우저에서 성능 데이터를 수집하는 방식입니다. Google의 `web-vitals` 라이브러리를 사용하면 Core Web Vitals를 클라이언트에서 수집할 수 있습니다.

```ts
import { onCLS, onINP, onLCP } from "web-vitals";

function sendToAnalytics(metric: unknown) {
  const body = JSON.stringify(metric);

  if (navigator.sendBeacon) {
    navigator.sendBeacon("/analytics/web-vitals", body);
    return;
  }

  fetch("/analytics/web-vitals", {
    method: "POST",
    body,
    keepalive: true,
  });
}

onCLS(sendToAnalytics);
onINP(sendToAnalytics);
onLCP(sendToAnalytics);
```

단순히 metric 값만 보내면 원인 분석이 어렵습니다. 가능하면 다음 정보도 함께 수집해야 합니다.

| 수집 정보            | 이유                                 |
| -------------------- | ------------------------------------ |
| route 또는 page type | 어떤 화면에서 문제가 발생하는지 파악 |
| device type          | 모바일과 데스크톱 차이 확인          |
| connection type      | 느린 네트워크 사용자 분리            |
| metric attribution   | 어떤 요소나 상호작용이 문제인지 추적 |
| app version          | 배포 후 회귀(regression) 여부 확인   |

> **주의**
> RUM을 설계할 때는 개인정보를 수집하지 않도록 주의해야 합니다. URL 쿼리, 입력값, 사용자 식별자 같은 민감 정보가 성능 로그에 섞이지 않도록 필터링해야 합니다.

## 지표별 개선 방향 요약

| 지표 | 지표가 나쁠 때 의심할 수 있는 병목 | 대표 개선 방향                                            |
| ---- | ---------------------------------- | --------------------------------------------------------- |
| TTFB | 문서 응답 시작 지연                | 서버 처리 최적화, CDN 캐시, 리다이렉트 제거               |
| FCP  | 흰 화면이 오래 지속됨              | 렌더링 차단 CSS 제거, critical CSS, 폰트 전략 개선        |
| LCP  | 주요 콘텐츠 렌더링 지연            | LCP 이미지 최적화, preload, fetch priority, SSR/캐시 개선 |
| CLS  | 예기치 않은 레이아웃 이동          | 이미지 크기 지정, 영역 예약, 폰트 로딩 전략 개선          |
| TBT  | 초기 로드 중 메인 스레드 차단      | JS 번들 축소, 코드 분할, 긴 작업 분리                     |
| TTI  | 상호작용 가능 상태 도달 지연       | 초기 JS 실행 비용 감소, hydration 비용 개선               |
| INP  | 입력 처리와 다음 페인트 지연       | 이벤트 핸들러 최적화, 렌더링 범위 축소, transition 적용   |

## 측정 루틴 예시

실무에서 사용할 수 있는 성능 측정 루틴은 다음과 같습니다.

1. **production 빌드 기준으로 Lighthouse를 실행합니다.**
   개발 서버는 번들 최적화, minification, 캐시 조건이 실제 배포와 다르기 때문에 성능 측정 기준으로 적합하지 않습니다.

2. **PageSpeed Insights에서 필드 데이터와 Lab 데이터를 분리해서 봅니다.**
   위쪽의 실제 사용자 데이터와 아래쪽의 Lighthouse 진단 결과는 역할이 다릅니다.

3. **LCP 요소와 TBT 원인을 DevTools Performance 패널에서 확인합니다.**
   점수만 확인하지 않고 어떤 리소스, 어떤 작업, 어떤 컴포넌트가 영향을 주는지 확인합니다.

4. **주요 사용자 플로우를 직접 수행하며 INP 후보를 찾습니다.**
   필터 클릭, 검색 입력, 모달 열기, 폼 제출처럼 사용자가 자주 하는 행동을 측정합니다.

5. **RUM으로 p75 변화를 추적합니다.**
   배포 전후 Lighthouse 점수가 개선되었더라도, 실제 사용자의 p75 LCP와 INP가 개선되지 않았다면 사용자 영향은 제한적일 수 있습니다.

## 결론

성능 측정은 점수를 올리는 작업이 아니라 사용자 시나리오에서 발생하는 대기 시간과 입력 지연을 줄이는 작업입니다.

Lighthouse는 초기 진단 도구로 사용할 수 있습니다. 하지만 Lighthouse가 보여주는 숫자는 통제된 환경의 단일 측정 결과입니다. 실제 사용자 경험을 개선하려면 필드 데이터로 문제를 발견하고, Lab 도구로 원인을 분석하고, 다시 RUM과 Core Web Vitals p75로 개선 여부를 검증해야 합니다.

정리하면 다음 원칙을 기억하면 됩니다.

1. FCP는 첫 콘텐츠 렌더링, LCP는 주요 콘텐츠 렌더링, CLS는 시각적 안정성, INP는 입력 반응성을 설명합니다.
2. TBT는 Lighthouse에서 중요한 지표지만 INP의 완전한 대체재는 아닙니다.
3. TTI는 참고할 수 있지만 최신 사용자 경험 판단에서는 LCP, INP, CLS 같은 Core Web Vitals가 더 중요합니다.
4. Lighthouse 점수는 문제 발견과 디버깅에 사용하고, 실제 개선 여부는 필드 데이터와 RUM으로 확인해야 합니다.
5. 평균보다 p75를 보고, URL보다 사용자 플로우를 기준으로 측정해야 합니다.
